# 📘 Part 1 --- Stack vs Heap
## 📝 Task 1: Trace the Memory
``` java
public class MemoryTest1 {
    public static void main(String[] args) {
        int x = 10;
        String name = "Danish";
        Student s1 = new Student("Ali", 20);

        modify(x, s1);

        System.out.println("x = " + x);
        System.out.println("Student age = " + s1.age);
    }

    static void modify(int x, Student s) {
        x = 50;
        s.age = 99;
    }
}

class Student {
    String name;
    int age;

    Student(String name, int age) {
        this.name = name;
        this.age = age;
    }
}
```
### 🔎 Questions
1.)Where is x stored? stack
2.)Where is s1 stored? stack
3.)Where is the Student object stored? heap
4.)Why does x remain 10 after modify()? A copy of the value of x is passed because it is a primitive variable
5.)Why does s1.age change? The value being passed is the memory address/reference of the object, so it is changing the reference itself

# 📘 Part 2 --- Object References and Aliasing
## 📝 Task 2: Reference Copying
``` java
public class MemoryTest2 {
    public static void main(String[] args) {
        Student s1 = new Student("Sara", 22);
        Student s2 = s1;

        s2.age = 30;

        System.out.println(s1.age);
    }
}
```
### 🔎 Questions
1.  How many objects are created? One object
2.  How many references exist? Two references exist, one for s1 and another for s2
3.  Why does modifying `s2` affect `s1`? Because when you create s2, Both s1 and s2 now hold the exact same memory address, so you modify the same address
### 🎯 Deliverable
Explain in 5--7 sentences how reference copying works in Java.
In Java, all arguments are passed by value, including object references. When you pass an object like Student s1 to a method, Java copies the reference (the memory address), 
not the actual object itself. This means the original variable and the method parameter both point to the same object in the heap. If the method changes the object’s 
internal state (for example, s.age = 99), the change affects the original object because there is only one copy of it in memory. However, if the method reassigns the reference itself 
(for example, s = new Student(...)), that reassignment only affects the local copy inside the method. The original reference in main will still point to the original object. 
This is why Java is described as pass-by-value of the reference, not pass-by-reference.
# 📘 Part 3 --- Garbage Collection
## 📝 Task 3: Eligible for Garbage Collection
``` java
public class MemoryTest3 {
    public static void main(String[] args) {
        Student s1 = new Student("John", 18);
        Student s2 = new Student("Emma", 19);

        s1 = s2;
    }
}
```
### 🔎 Questions
1.  How many objects are created?  Two objects are created, student object with john and 18, the other with emma and 19
2.  Which object becomes eligible for garbage collection? The student object with john and 18, because when you do s1=s2 you overwrite the address in s1 with the address for s2,
   then the john and 18 become unreachable
4.  At what line does that happen? It happens at the line s1=s2
5.  Does Java immediately delete it? No, the JVM decides when to delete

# 📘 Part 4 --- Memory Leak Simulation
## 📝 Task 4: Static Reference Leak
``` java
import java.util.ArrayList;

public class MemoryLeakExample {

    static ArrayList<Student> students = new ArrayList<>();

    public static void main(String[] args) {

        for(int i = 0; i < 100000; i++) {
            students.add(new Student("Student" + i, i));
        }

        System.out.println("Done creating students");
    }
}
```
### 🔎 Questions
1.  Why are these objects NOT garbage collected?
The students ArrayList is declared as static. In Java, static fields are associated with the Class object rather than a specific instance.
Since the Class object itself is typically loaded for the entire lifetime of the application,
any object it references is considered "reachable" by the Garbage Collector
Even after the for loop finishes and the main method reaches its end, the students liststill holds a reference to every single Student object.
Because the list is still reachable via the static root, the GC cannot reclaim that memory.
2.  What would you change to fix this?
Explicitly call students.clear() or set students = null after the processing is done.
3.  Modify the code to prevent the leak.
```java
import java.util.ArrayList;

public class MemoryLeakExample {

    // Removed the static class-level variable

    public static void main(String[] args) {
        // Scope the list to the main method
        ArrayList<Student> students = new ArrayList<>();

        for(int i = 0; i < 100000; i++) {
            students.add(new Student("Student" + i, i));
        }

        System.out.println("Done creating students");
        
        // Once main finishes, 'students' goes out of scope 
        // and all 100,000 objects become eligible for GC.
    }
}
```
# 📘 Part 5 --- Forcing Memory Pressure
## 📝 Task 5: OutOfMemoryError
``` java
import java.util.ArrayList;

public class MemoryPressure {
    public static void main(String[] args) {

        ArrayList<int[]> list = new ArrayList<>();

        while(true) {
            list.add(new int[1000000]);
        }
    }
}
```
### 🔎 Questions
1.  What exception occurs? The program will eventually throw a java.lang.OutOfMemoryError: Java heap space.
2.  Why does it happen?
Because the while(true) loop never terminates, continuously creating new objects.
Every time a new int[] array is created, it is added to the list ArrayList.
Because the list stays in scope, it maintains a strong reference to every array ever created.
Since the Garbage Collector (GC) can only reclaim objects that are no longer reachable, and every array is still "reachable" via the list,
the GC cannot free any memory. Eventually, the size of the combined arrays exceeds the maximum memory allocated to the JVM.
4.  What JVM argument can limit heap size?
You can use the -Xmx flag to set the maximum heap size.
Try running with:

    -Xmx128m

# 📘 Part 6 --- Reflection / Conceptual Questions

1.  Why doesn't Java allow manual memory deallocation?
Java removes manual deallocation (like free() in C or delete in C++) primarily for safety and simplicity.
Manual deallocation often leads to "Dangling Pointers" (trying to use memory that has already been freed) or
"Double Frees" (trying to free the same memory twice), both of which cause crashes or security vulnerabilities.
By handling memory automatically, Java allows developers to focus on business logic rather than the tedious plumbing of
tracking every object’s lifecycle.
2.  What is the advantage of garbage collection?
The biggest advantage is automated lifecycle management. It significantly reduces memory leaks and memory-related bugs.
The Garbage Collector (GC) is "smart"—it knows exactly when an object is no longer reachable by any part of the program
 and reclaims that space without the developer having to write a single line of cleanup code.
3.  What is the trade-off compared to C++?
While Java is "safer," it pays a price in performance and control. The GC is a background process that consumes CPU cycles.
In C++, you have zero overhead because you control exactly when memory is released.
Java typically uses more RAM because the GC doesn't always reclaim memory the exact millisecond it's no longer needed,
it waits for an optimal time to run.
4.  What is a "strong reference"?
As long as an object has at least one strong reference pointing to it, it is ineligible for garbage collection.

    double balance;

        balance = initialBalance;

        balance += amount;
